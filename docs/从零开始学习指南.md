## 2. 理解架构

### 2.1 这个系统做什么？

```
用户输入数据 → 系统清洗处理 → AI模型总结 → 返回结果
```

一句话：**把任何数据（文本、网页、CSV）扔进来，自动清洗、分段，然后用 AI 总结出要点。**

### 2.2 为什么要分 5 个模块？

```
agent-core      ← 接口定义（合约）
agent-data      ← 数据接入（适配器）
agent-pipeline  ← 数据处理（流水线）
agent-model     ← AI 推理（模型）
agent-server    ← Web 服务（入口）
```

**核心思想：依赖倒置。** 上层依赖接口，不依赖实现。

```
agent-server 依赖 agent-core（接口）
agent-server 不直接依赖 agent-data/agent-pipeline/agent-model（实现）

Spring 在运行时把实现注入进来 —— 这就是 DI（依赖注入）的威力。
```

### 2.3 模块依赖关系

```
agent-server ──→ agent-core
     │               ↑
     ├──→ agent-data ──┤
     ├──→ agent-pipeline ──┤
     └──→ agent-model ──┘
```

所有模块都依赖 `agent-core`（接口），但互不依赖。加新功能只需加新模块，不改旧代码——**开闭原则**。

## 3. 核心抽象：接口即合约

> 📂 代码位置：`agent-core/src/main/java/com/learner/agent/core/`

### 3.1 Agent —— 系统的"大脑"

```java
// 文件：Agent.java
public interface Agent {
    TaskResult execute(Task task);                    // 同步执行
    CompletableFuture<TaskResult> executeAsync(Task task); // 异步执行
    TaskResult getTaskStatus(String taskId);          // 查状态
    boolean cancelTask(String taskId);                // 取消任务
    String quickSummarize(String text, int maxLen, int minLen); // 快速总结
}
```

**设计要点：**
- 接口只定义"做什么"，不定义"怎么做"
- `executeAsync` 返回 `CompletableFuture`，不阻塞调用者
- `quickSummarize` 是最常用的入口，单独列出

### 3.2 Task —— 描述一次任务

```java
// 文件：Task.java
public class Task {
    private String id = UUID.randomUUID().toString().substring(0, 8);
    private TaskType type;        // FULL_PIPELINE / SUMMARIZE_ONLY / PROCESS_ONLY
    private TaskStatus status;    // PENDING → RUNNING → COMPLETED / FAILED
    private String sourceType;    // "text" / "json" / "csv" / "web"
    private Map<String, Object> sourceParams; // 具体数据（文本内容、URL等）
    private SummarizeOptions summarizeOptions; // 总结参数
}
```

**设计要点：**
- ID 自动生成，调用者不用操心
- 状态机模式：PENDING → RUNNING → COMPLETED/FAILED/CANCELLED
- `sourceParams` 是 `Map`，支持任意数据格式——这就是"通用"的秘诀

### 3.3 Pipeline + Processor —— 流水线

```java
// 文件：Pipeline.java
public interface Pipeline {
    ProcessedData execute(ProcessedData input, Map<String, Object> config);
    Pipeline addProcessor(Processor processor);  // 链式调用
}
```

```java
// 文件：Processor.java
public interface Processor {
    ProcessedData process(ProcessedData input, Map<String, Object> config);
}
```

**这就是责任链模式的 Java 实现。** 每个 Processor 干一件事，Pipeline 负责串起来。

### 3.4 DataSource —— 数据源

```java
// 文件：DataSource.java
public interface DataSource<T> {
    String getName();              // 如 "text", "web"
    boolean supports(String type); // 如 type="csv" → CsvDataSource 返回 true
    ProcessedData fetch(Map<String, Object> params); // 获取数据，统一返回 ProcessedData
}
```

**策略模式：** 不同数据源是不同策略，运行时根据 type 选择。

### 3.5 Summarizer —— 总结器

```java
// 文件：Summarizer.java
public interface Summarizer {
    String summarize(String text, SummarizeOptions options);
    List<String> summarizeBatch(List<String> texts, SummarizeOptions options);
    boolean isReady(); // 模型加载好了吗？
}
```

**把 AI 能力抽象成一个接口。** 换模型只需换实现，上层代码不动。

## 4. 数据接入：适配器模式

> 📂 代码位置：`agent-data/src/main/java/com/learner/agent/data/`

### 4.1 四种数据源

| 类 | 处理什么 | 核心逻辑 |
|----|----------|----------|
| `TextDataSource` | 直接文本 | 直接返回 `new ProcessedData(text)` |
| `JsonDataSource` | JSON | 解析 JSON，提取指定字段或全部拼接 |
| `CsvDataSource` | CSV | 用 Hutool 解析，转成 Markdown 表格 |
| `WebDataSource` | 网页 | 用 Jsoup 抓取 + 提取正文 |

### 4.2 TextDataSource 源码精读

```java
public class TextDataSource implements DataSource<ProcessedData> {

    @Override
    public String getName() { return "text"; }

    @Override
    public boolean supports(String type) {
        return "text".equalsIgnoreCase(type) || "txt".equalsIgnoreCase(type);
    }

    @Override
    public ProcessedData fetch(Map<String, Object> params) {
        String text = (String) params.getOrDefault("text", "");
        if (text == null || text.isBlank()) {
            throw new IllegalArgumentException("text 参数不能为空");
        }
        return new ProcessedData(
            "txt-" + System.currentTimeMillis(),
            text,
            "direct-input",
            "text"
        );
    }
}
```

**关键点：**
- `supports()` 判断这个数据源能不能处理
- `fetch()` 入参是 `Map`，不管上游传什么都能接
- 返回统一类型 `ProcessedData`，下游处理器不用关心来源

### 4.3 怎么加新数据源？三步

假设你要加 PDF 支持：

```java
// 第1步：实现接口
public class PdfDataSource implements DataSource<ProcessedData> {
    @Override
    public String getName() { return "pdf"; }

    @Override
    public boolean supports(String type) { return "pdf".equalsIgnoreCase(type); }

    @Override
    public ProcessedData fetch(Map<String, Object> params) {
        byte[] pdfBytes = (byte[]) params.get("bytes");
        // 用 PDFBox 提取文字...
        String text = extractText(pdfBytes);
        return new ProcessedData("pdf-" + System.currentTimeMillis(), text, "pdf-file", "pdf");
    }
}

// 第2步：在 AgentService 中注册
this.dataSources.add(new PdfDataSource());

// 第3步：调用
POST /api/v1/agent/process
{ "type": "pdf", "params": { ... } }
```

**不改核心代码，只加一个类——这就是开闭原则。**

## 5. 流水线：责任链模式

> 📂 代码位置：`agent-pipeline/src/main/java/com/learner/agent/pipeline/`

### 5.1 流水线的执行过程

```
原始文本
  ↓
CleanProcessor    ← 去噪、统一格式
  ↓
ChunkProcessor    ← 切成小块（适配模型 token 限制）
  ↓
[对每块调用 Summarizer]
  ↓
AggregateProcessor ← 合并各块结果
  ↓
最终结果
```

### 5.2 DefaultPipeline 源码精读

```java
public class DefaultPipeline implements Pipeline {
    private final List<Processor> processors = new ArrayList<>();

    @Override
    public DefaultPipeline addProcessor(Processor processor) {
        this.processors.add(processor);
        return this;  // 返回 this，支持链式调用
    }

    @Override
    public ProcessedData execute(ProcessedData input, Map<String, Object> config) {
        ProcessedData current = input;
        for (Processor processor : processors) {
            long start = System.currentTimeMillis();
            current = processor.process(current, config);
            log.debug("{} 完成 ({}ms)", processor.getName(),
                    System.currentTimeMillis() - start);
        }
        return current;
    }
}
```

**关键点：**
- `addProcessor` 返回 `this` → **链式调用**：`pipeline.add(a).add(b).add(c)`
- `execute` 是**同步循环**，一个处理完才到下一个
- 每个 Processor 的输入是上一个的输出 → **数据单向流动**

### 5.3 CleanProcessor —— 为什么需要清理？

```java
// 真实世界的文本有很多"脏东西"：
String dirty = "Hello  <br>  World\r\n\r\n\r\n  \"测试\"  ";

// CleanProcessor 处理后：
String clean = "Hello World\n\n\"测试\"";
// HTML 标签没了、换行统一了、多余空格没了、引号标准化了
```

### 5.4 ChunkProcessor —— 为什么要分段？

```
Transformer 模型有 token 限制（通常 512 或 1024 token）
一篇 10000 字的文章 → 需要切成约 10 块 → 分别总结 → 最后合并
```

**分段策略（三级降级）：**
1. 先按段落分（最自然）
2. 段落太长按句子分
3. 句子还太长硬截断

## 6. 模型推理：DJL 实战

> 📂 代码位置：`agent-model/src/main/java/com/learner/agent/model/`

### 6.1 两种总结模式

```
模型加载成功 → 生成式总结（AI 重新组织语言）
模型未就绪   → 抽取式总结（选原文关键句拼起来）
```

**抽取式总结算法（TF-ISF）：**
```
1. 把文章拆成句子
2. 每个句子打分 = 句内关键词的 TF × ISF
3. 选得分最高的 1/3 句子
4. 按原文顺序拼成摘要
5. 第一句和最后一句加分（位置权重）
```

### 6.2 ModelManager —— 模型生命周期

```java
ModelManager 管理四个状态：

REGISTERED → LOADING → READY
                         ↓
                       ERROR
```

**关键方法：**
- `register()` — 注册模型信息（不加载）
- `loadPredictor()` — 加载模型文件，返回可推理的 Predictor
- `close()` — 释放所有模型资源

### 6.3 SummarizationTranslator —— 数据转换

```
输入文本 → Tokenize → NDArray（模型能懂的格式） → 推理
模型输出（数字） → Detokenize → 输出文本
```

`Translator` 是 DJL 的核心概念：**把人类语言和模型张量互相转换。**

### 6.4 怎么下载真正的 AI 模型？

抽取式总结只是后备方案。要体验真正的 AI 总结：

#### 方式一：下载 DistilBART（推荐入门）

```bash
# 1. 安装 HuggingFace CLI
pip install huggingface_hub

# 2. 下载 DistilBART-CNN 模型（~300MB）
huggingface-cli download sshleifer/distilbart-cnn-6-12 --local-dir ./models/distilbart-cnn-6-12

# 3. 转换为 ONNX 格式（需要安装 optimum）
pip install optimum onnx onnxruntime
optimum-cli export onnx --model ./models/distilbart-cnn-6-12 ./models/distilbart-cnn-6-12-onnx

# 4. 把 ONNX 文件放到 models/ 目录下
# 5. 重启服务，调用 POST /api/v1/models/load { "name": "distilbart-cnn-6-12" }
```

#### 方式二：下载 T5-Small（更轻量，~240MB）

```bash
huggingface-cli download google-t5/t5-small --local-dir ./models/t5-small
optimum-cli export onnx --model ./models/t5-small ./models/t5-small-onnx
```

#### 方式三：下载 Qwen2.5（中文最佳，但需要 GPU）

阿里通义千问小模型，中文效果最好，但 1.5B 版本需要约 3GB 显存。

## 7. REST API：把能力暴露出去

> 📂 代码位置：`agent-server/src/main/java/com/learner/agent/server/`

### 7.1 完整 API 列表

| 方法 | 路径 | 功能 | 同步/异步 |
|------|------|------|-----------|
| POST | `/api/v1/agent/summarize` | 直接总结 | 同步 |
| POST | `/api/v1/agent/process` | 完整处理任务 | 异步 |
| GET | `/api/v1/agent/task/{id}` | 查任务结果 | 同步 |
| GET | `/api/v1/models` | 列出模型 | 同步 |
| POST | `/api/v1/models/load` | 加载模型 | 同步 |

### 7.2 统一响应格式

```json
{
  "code": 200,
  "message": "success",
  "data": { ... },
  "timestamp": 1782609218391
}
```

所有 API 返回这个格式，前端处理统一。

### 7.3 AgentService —— 业务编排

```java
// 这是整个系统的核心编排逻辑：
public TaskResult execute(Task task) {
    // 1. 根据 type 选择数据源
    ProcessedData data = fetchData(task);

    // 2. 走流水线（清理 + 分段）
    data = runPipeline(data, task.getPipelineName());

    // 3. 对每个分段调用 AI 总结
    List<String> chunkSummaries = data.getChunks().stream()
        .map(chunk -> summarizer.summarize(chunk, options))
        .toList();

    // 4. 合并返回
    String summary = String.join("\n\n", chunkSummaries);
    return TaskResult.success(task.getId(), summary, ...);
}
```

### 7.4 异步任务怎么工作的？

```
用户 POST /api/v1/agent/process
  → Controller 创建 Task，调用 agent.executeAsync(task)
  → AgentService 在线程池中异步执行
  → 立即返回 202 Accepted + taskId
  → 用户轮询 GET /api/v1/agent/task/{id}
  → 完成时返回 COMPLETED + 结果
```

### 7.5 全局异常处理

```java
@RestControllerAdvice
public class GlobalExceptionHandler {
    // 参数错误 → 400
    // 模型未就绪 → 503
    // 未知异常 → 500（带日志）
}
```

**所有异常统一捕获，API 永远不会返回 500 的 Tomcat 白页。**

## 附录：关键设计模式总结

| 模式 | 在本项目哪里 | 解决了什么问题 |
|------|-------------|---------------|
| **策略模式** | `DataSource` 接口 + 多个实现 | 不同数据源切换不改代码 |
| **责任链模式** | `Pipeline` + `Processor` | 处理步骤可自由组合 |
| **模板方法** | `Agent.execute()` | 定义流程骨架，具体步骤可替换 |
| **适配器模式** | `DataSource` | 把各种格式统一成 `ProcessedData` |
| **工厂模式** | `Task.ofText()` / `TaskResult.success()` | 简化对象创建 |
| **依赖注入** | Spring `@Bean` + 构造器注入 | 解耦组件，方便测试 |

---

> 💡 **核心心得：好的架构不是一次性设计出来的，而是不断"发现"出来的。**
> 每次当你发现自己在不同的地方写相似的代码，那就是一个抽象的机会。
> 抽象不是目的，消除重复、降低耦合才是。
